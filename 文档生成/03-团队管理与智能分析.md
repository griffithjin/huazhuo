## 1. 团队管理计划

### 1.1 组织架构设计

寰卓 采用**扁平化+矩阵式**组织架构，初期（50人以内）设6大核心部门，后续随业务扩展增设风控、合规、法务等中台部门。

#### 1.1.1 组织架构图

```
                    ┌─────────────┐
                    │    CEO/创始人  │
                    │  (战略+融资+品牌) │
                    └──────┬──────┘
                           │
        ┌──────────────────┼──────────────────┐
        │                  │                  │
   ┌────┴────┐        ┌────┴────┐        ┌────┴────┐
   │   CTO   │        │   COO   │        │   CMO   │
   │ (技术架构)│        │ (运营增长)│        │ (市场品牌)│
   └────┬────┘        └────┬────┘        └────┬────┘
        │                  │                  │
   ┌────┼────┐        ┌────┼────┐        ┌────┼────┐
   │    │    │        │    │    │        │    │    │
┌──┴┐ ┌┴┐ ┌─┴─┐   ┌──┴┐ ┌┴┐ ┌─┴─┐   ┌──┴┐ ┌┴┐ ┌─┴─┐
│后端│ │AI│ │前端│   │运营│ │客服│ │风控│   │市场│ │渠道│ │内容│
│开发│ │组│ │开发│   │增长│ │支持│ │合规│   │品牌│ │代理│ │营销│
└───┘ └──┘ └───┘   └───┘ └──┘ └───┘   └───┘ └──┘ └───┘
```

#### 1.1.2 部门职能与编制（Phase 1: 0-6个月）

| 部门 | 负责人职级 | 编制人数 | 核心职能 |
|------|-----------|---------|---------|
| **CEO办公室** | CEO（创始人） | 1-2人 | 战略决策、融资路演、政府关系、重大合作 |
| **技术部** | CTO | 8-12人 | 平台开发、API网关、AI模型、安全架构、DevOps |

#### 1.2.4 技术团队详细岗位

**后端开发工程师（API/交易引擎）**

| KPI指标 | 定义 | 计算公式 | 目标值 |
|---------|------|---------|--------|
| API响应时间(P99) | 99%请求的响应耗时 | `percentile(response_time, 99)` | ≤200ms |
| 接口可用性 | API正常运行时间占比 | `uptime / total_time × 100%` | ≥99.95% |
| 代码质量评分 | SonarQube综合评分 | 代码覆盖率×0.3 + 漏洞数×0.3 + 规范×0.4 | ≥85分 |
| 事故修复时间(MTTR) | 故障发生到恢复平均时间 | `Σ(修复时间) / 事故次数` | ≤30分钟 |

**AI工程师（风控/推荐/预测）**

| KPI指标 | 定义 | 计算公式 | 目标值 |
|---------|------|---------|--------|
| 风控模型准确率 | 识别欺诈交易的准确度 | `(TP+TN)/(TP+TN+FP+FN)` | ≥98% |
| 推荐点击率 | AI推荐内容的CTR | `点击数 / 曝光数 × 100%` | ≥5% |
| 预测准确率 | 收入/用户预测准确度 | `1 - |预测值-实际值|/实际值` | ≥85% |
| 模型推理延迟 | 单次预测耗时 | 端到端推理时间 | ≤100ms |

**DevOps/SRE工程师**

| KPI指标 | 定义 | 计算公式 | 目标值 |
|---------|------|---------|--------|
| 部署频率 | 日均生产部署次数 | `成功部署次数 / 工作日数` | ≥5次/天 |
| 变更失败率 | 导致事故的部署占比 | `失败部署 / 总部署 × 100%` | ≤2% |
| 基础设施成本 | 每万笔交易的云成本 | `月度云费用 / (交易量/10000)` | 月降5% |
| 安全事件数 | 安全漏洞与入侵事件 | 统计周期内事件数 | 0 |

### 3.3 收入分析

#### 3.3.1 核心收入指标

| 指标 | 英文缩写 | 定义 | 计算公式 | 用途 |
|------|---------|------|---------|------|
| **月度经常性收入** | MRR | 可预测的月度订阅/重复收入 | `Σ(订阅用户月费) + 预估重复购买` | SaaS/订阅业务 |
| **年度经常性收入** | ARR | MRR × 12 | `MRR × 12` | 融资估值、年度规划 |
| **总收入** | GMV | 平台总交易额 | `Σ(订单金额)` | 平台规模指标 |
| **净收入** | Net Revenue | 扣除退款/手续费后收入 | `GMV - 退款 - 手续费 - 佣金` | 真实收入 |
| **客单价** | AOV | 平均订单金额 | `GMV / 订单数` | 定价策略 |
| **每用户平均收入** | ARPU | 每用户平均贡献收入 | `月度收入 / MAU` | 变现效率 |
| **每付费用户收入** | ARPPU | 每付费用户平均收入 | `月度收入 / 付费用户数` | 核心用户价值 |
| **用户终身价值** | LTV | 用户全生命周期总价值 | `ARPU × 平均生命周期(月) × 毛利率` | 获客预算 |
| **获客成本** | CAC | 获取一个用户的平均成本 | `总营销费用 / 新增用户数` | 效率指标 |
| **LTV/CAC** | 投入产出比 | 用户价值/获客成本 | `LTV / CAC` | ≥3:1健康 |
| **回收周期** | Payback | CAC回收所需月数 | `CAC / (ARPU × 毛利率)` | ≤12月健康 |

#### 3.3.2 收入构成分析

| 维度 | 细分项 | 计算公式 | 分析目的 |
|------|--------|---------|---------|
| **地域分布** | 北美/欧洲/亚太/拉美/非洲 | `区域GMV / 总GMV × 100%` | 市场重心 |
| **渠道分布** | 直营/代理/白标/API | `渠道GMV / 总GMV × 100%` | 渠道效率 |
| **用户分层** | 新客/老客/核心/VIP | `分层GMV / 总GMV × 100%` | 用户价值 |
| **Token类型** | USDT/BTC/ETH/其他 | `类型GMV / 总GMV × 100%` | 产品策略 |
| **订单大小** | <$50/$50-200/$200-1K/>$1K | `区间订单数 / 总订单数 × 100%` | 定价优化 |
| **付费周期** | 首次/复购/3次+/订阅 | `周期订单数 / 总订单数 × 100%` | 复购健康 |

#### 3.3.3 收入预测模型

| 模型 | 适用场景 | 输入变量 | 输出 |
|------|---------|---------|------|

```python
# 佣金计算伪代码
def calculate_commission(agent_id, month):
    orders = get_agent_orders(agent_id, month)
    
    # 基础佣金
    base_commission = sum(order.amount * agent.discount_rate 
                         for order in orders)
    
    # 阶梯加成
    total_gmv = sum(order.amount for order in orders)
    if total_gmv >= agent.tier.target * 2:  # 超额200%
        bonus_rate = 0.05
    elif total_gmv >= agent.tier.target * 1.5:  # 超额150%
        bonus_rate = 0.03
    elif total_gmv >= agent.tier.target * 1.2:  # 超额120%
        bonus_rate = 0.02
    else:
        bonus_rate = 0
    
    bonus = total_gmv * bonus_rate
    
    # 下级代理分润（仅一级/二级）
    sub_commission = 0
    for sub in agent.subordinates:
        sub_orders = get_agent_orders(sub.id, month)
        sub_gmv = sum(o.amount for o in sub_orders)
        sub_commission += sub_gmv * agent.tier.sub_bonus_rate
    
    # 特殊激励
    special_bonus = calculate_special_bonus(agent_id, month)
    
    total = base_commission + bonus + sub_commission + special_bonus
    
    return {
        'base': base_commission,
        'bonus': bonus,
        'sub_commission': sub_commission,
        'special': special_bonus,
        'total': total
    }
```

## 5. 商业化KPI体系与增长路径

### 5.1 北极星指标

| 指标 | 定义 | 为什么选它 | 计算公式 |
|------|------|----------|---------|
| **周活跃付费用户数 (WAPU)** | 近7天有交易的付费用户数 | 同时反映活跃、付费、频次 | `count(distinct user_id where order_date >= today-7)` |
| 备选：月活跃交易用户 (MATU) | 近30天有交易的用户 | 更稳定的长期指标 | `count(distinct user_id where order_date >= today-30)` |

**北极星指标拆解树**

```
WAPU（周活跃付费用户）
├── 新增付费用户
│   ├── 流量 × 注册率 × KYC率 × 首充率 × 首购率
│   └── 各渠道流量 × 各渠道转化率
├── 老客复购
│   ├── 活跃老客数 × 购买频次
│   └── 分层运营（核心/活跃/潜力/沉默）
└── 流失召回
    ├── 流失用户池 × 召回触达率 × 召回转化率
    └── 召回渠道（邮件/短信/APP Push/代理）
```

| CAC | `总营销费用 / 新增用户数` |

| 客单价 | `GMV / 订单数` |
| 复购率 | `复购用户数 / 总购买用户数 × 100%` |
| 库存周转 | `平均库存价值 / 日均销售成本` |
| 代理佣金 | `GMV × 代理折扣率 + 阶梯奖金 + 下级分润` |
| 毛利率 | `(收入 - 成本) / 收入 × 100%` |
